# 한국어 PII NER 라벨 가이드 v1

> **목적**: NER 학습 데이터 라벨링의 일관성 확보. 모든 annotation·합성 데이터 생성·매핑 룰이 이 문서를 따른다.
> **작성**: 2026-05-08, NER Owner / **승인 필요**: Pipeline Owner + 임정묵 교수

## 0. 라벨 클래스 (3개만)

NER 학습은 **NAME / ADDRESS / ORG** 3개 클래스만. 나머지(TITLE/DEPT/DISEASE/CONTACT)는 사전·정규식이 담당하므로 라벨링 X.

| 클래스 | 정의 | KLUE-NER 매핑 |
|---|---|---|
| **NAME** | 사람 이름 (실명·예명·핸들 포함) | PER → NAME |
| **ADDRESS** | 주소 (행정구역+도로명+동호수까지 포함하는 full span) | LOC → ADDRESS |
| **ORG** | 조직 (회사·학교·병원·기관·정부) | OG/ORG → ORG |

---

## 1. NAME 경계 정의

### ✅ 포함
- 단순 이름: `홍길동`, `Mike`
- 영문 이름: `John Smith`
- 핸들/별명이 명시적 PII 컨텍스트: `(고객 닉네임 박사장)`의 `박사장`

### ❌ 제외 (NAME으로 라벨링 X)
- 공인 인물 (역사·정치인): `이순신`, `세종대왕` — 일반명사 취급
- 직책 단독: `과장`, `팀장`, `원장` — TITLE 클래스 (사전)
- 호칭 단독: `님`, `씨`, `선배`, `학생`

### 🤔 경계 케이스 (꼭 합의)

| 입력 | 라벨 | 이유 |
|---|---|---|
| `박과장` | **NAME ("박") + TITLE ("과장")** 분리 | 이름+직책은 분리. NER이 박을 NAME 후보, 직책은 사전 |
| `홍길동님` | **NAME ("홍길동")**, "님"은 suffix 보정 | 호칭은 boundary corrector 분리 |
| `김 과장` | **NAME ("김") + TITLE ("과장")** | 띄어쓰기 있어도 동일 정책 |
| `진아야` | **NAME ("진아")**, "야"는 호격 suffix | 호격 어미 분리 |
| `홍길동이` | **NAME ("홍길동")**, "이"는 조사 suffix | 조사 분리 (Boundary Corrector 책임) |
| `사장님` | **TITLE만** (NAME 없음) | 직책 단독은 NAME X |
| `박사장` (직함 vs 이름 불명확) | **NAME ("박사장")** if 컨텍스트 PII | 모호하면 컨텍스트로 판정 |
| `김민수 학생` | **NAME ("김민수")**, "학생"은 TITLE | 분리 |
| `세종대왕` | **라벨 X** (공인) | 일반명사 취급 |
| `이순신 장군` | **라벨 X** (공인) | 동일 |
| 외국 이름 `Mike` | **NAME** | 영문 이름 포함 |

## 2. ADDRESS 경계 정의

### ✅ 포함 (full span 우선)
- 행정구역: `서울특별시`, `강남구`, `역삼동`
- 도로명: `테헤란로 152`
- 지번: `123-45`
- 동/호: `101동 1203호`
- **완전 주소는 가장 긴 span**: `서울특별시 강남구 테헤란로 152 101동 1203호` 통째로 ADDRESS

### ❌ 제외
- 지명만 (장소 표현): `서울에 갔다`의 `서울` — 라벨 X 또는 짧은 ADDRESS
- 회사 위치 표현: `(주)삼성 본사` — ORG로 처리
- 일반 지리: `한반도`, `유럽`

### 🤔 경계 케이스

| 입력 | 라벨 | 이유 |
|---|---|---|
| `서울` (단독) | **ADDRESS (낮은 위험 P3)** 또는 라벨 X | 컨텍스트 보정 — "거주지 서울" → P2, "서울 날씨" → 라벨 X |
| `서울 강남구 역삼동` | **ADDRESS** (full span) | 3-level 주소는 full span |
| `테헤란로 152` | **ADDRESS** | 도로명+번지 |
| `101동 1203호` | **ADDRESS** | 동·호 단독도 ADDRESS_UNIT (high risk P1) |
| `서울특별시 강남구 테헤란로 152, 101동 1203호` | **ADDRESS 1개 (full)** | 콤마·공백 무관, 통째 |
| `우리 집` | **라벨 X** | 비식별 표현 |
| `강남역 근처 원룸 302호` | **ADDRESS** (보강 사전 location 카테고리) | 역세권+호수 — Tier 4 컨텍스트 |
| 회사 주소 `(주)삼성 본사 서울 강남` | **ORG ("(주)삼성 본사") + ADDRESS ("서울 강남")** 분리 | 두 entity 분리 |

---

## 3. ORG 경계 정의

### ✅ 포함
- 회사: `삼성전자`, `(주)네이버`, `카카오`
- 학교: `서울대학교`, `고려대 컴퓨터공학과`
- 병원: `세브란스병원`, `삼성서울병원`
- 정부·공공: `국세청`, `KOICA`, `한국전력공사`
- 부서·팀까지 포함하는 경우만 (단독 부서명은 DEPT 클래스):
  - `(주)삼성 인사팀` → ORG ("(주)삼성") + DEPT ("인사팀") 분리

### ❌ 제외
- 부서·팀 단독: `인사팀`, `개발팀` → **DEPT 클래스 (사전+정규식, NER 학습 X)**
- 직책 단독: `이사회`, `위원회` (모호) → 라벨 X 또는 컨텍스트 판정

### 🤔 경계 케이스

| 입력 | 라벨 | 이유 |
|---|---|---|
| `(주)삼성전자` | **ORG** ("(주)삼성전자") | 법인 표시 포함 |
| `삼성전자 인사팀 김과장` | **ORG ("삼성전자") + DEPT ("인사팀") + NAME ("김") + TITLE ("과장")** | 4개 분리 |
| `서울대 학생` | **ORG ("서울대")**, "학생"은 TITLE | 학교 + 직책 분리 |
| `한국은행` (금융기관 vs 일반명사) | **ORG** | 명백한 금융기관 |
| `당뇨협회` | **ORG** | 협회·단체 포함 |
| `회사` (단독) | **라벨 X** | 일반명사 |

## 4. 라벨링 우선순위 (모호 시)

1. **PII로 식별 가능 가중**: 모호하면 PII 쪽으로 (recall 우선)
2. **컨텍스트 우선**: 단독 어휘보다 주변 단어 (고객명·연락처·배송지)
3. **공인·역사 인물 제외**: NAME 라벨 X
4. **단독 일반명사 제외**: "회사", "동네" 같은 표현은 라벨 X

---

## 5. BIO 태그 규칙

NER 학습은 BIO 태깅 (Begin / Inside / Outside):

```
홍길동이      삼성전자에     입사했다
B-NAME I-NAME B-ORG  I-ORG  O      O      O
```

(조사 "이"는 학습 시 I-NAME 처리, 추론 후 Boundary Corrector가 분리)

**라벨 alignment 규칙**:
- subword 토큰화 시 **첫 subword만 라벨**, 나머지는 -100 (loss 무시)
- `김철수` → `김철`(B-NAME) + `##수`(-100)
- 추론 시 `word_ids()`로 같은 word_id 토큰에 라벨 propagate

## 6. raw PII 처리 원칙 (운영)

- 라벨링 데이터는 **합성 (Faker-ko) + 공개 (KLUE-NER)** 만 사용
- 실제 사용자 PII는 절대 학습/평가 데이터에 포함 X
- 평가 결과 보고 시 raw 텍스트 ❌, hash·offset·type만 ✅
