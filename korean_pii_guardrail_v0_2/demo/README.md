## 1. 실행

```powershell
cd korean_pii_guardrail_v0_2
pip install -r demo/requirements.txt   # 최초 1회 (gradio/matplotlib/transformers/torch)
$env:PYTHONUTF8 = "1"
python demo/app.py
# 브라우저에서 http://127.0.0.1:7860
```

- **첫 실행 시 NER v3 모델 로딩에 ~20초.** 로컬 학습 산출물(`PII/ner/models/pii_ner_v3/final`)이 있으면 그것을, 없으면 **HuggingFace Hub `vmaca123/korean-pii-ner-v3`에서 자동 다운로드**(1.3GB, 최초 1회)한다. 즉 clone 후 바로 실행 가능.
- real NER 로드 실패 시 자동으로 mock NER fallback (상단 배너에 모드 표시).
- 대시보드용 측정 데이터/그림은 `demo/assets/`에 동봉되어 별도 경로 설정 불필요.

## 3. 발표 흐름 (3막, 약 7분)

### Act 1 — 문제 제기 (말 + ③탭 1차 figures, ~1.5분)
> "프로덕션 PII 가드레일(Presidio 등)은 영어 중심이라 한국어, 특히 **변이 공격**에 약합니다. 1차 캡스톤에서 8,400건 퍼저로 우회율을 측정했고(③탭 그래프), 그 한계를 메우는 **단일 통합 가드레일**을 만들었습니다."

### Act 2 — 라이브 데모 (①탭, ~3분) ★ 핵심
> 교수님께 직접 타이핑 권하거나 예제 버튼 클릭:
> - `김민수 주민번호 900101-1234568, 서울시 강남구 테헤란로 123 거주`
> - → `[PERSON_1] 주민번호 [REDACTED], [ADDRESS_1]` 실시간 마스킹
>
> 강조 포인트:
> - **이름·주소**는 fine-tune한 NER v3가, **주민번호·전화·이메일·카드**는 정규식+체크섬이 잡음
> - 처리 시간 ~200ms (탐지 상세표에 위험도 P0/P1, 조치 mask 표시)

### Act 3 — 성능 + 방어 입증 (②③탭, ~2.5분)
> **②탭**: "각 단계가 성능에 기여하는 걸 ablation으로 분리 측정했고(그래프), NER를 ONNX INT8로 양자화해 **추론을 4배 빠르게(547ms→135ms)** 만들면서 정확도는 99.7% 유지했습니다."
>
> **③탭**: "핵심은 변이 방어입니다. `ㅈㅜㅁㅣㄴ번호`(자모분해)나 zero-width가 섞인 주민번호를 넣어도 — " (라디오 선택 → 처리 버튼) " — L0 정규화가 복원해서 그대로 탐지/마스킹합니다."

| 항목 | 값 |
|------|-----|
| NER v3 macro-F1 | **0.878** (internal test) / **0.766** (KLUE 외부) |
| NER 학습 | klue-roberta-large, 34k 데이터, $0.92, 30분 |
| INT8 가속 | **4.04x** (PyTorch 547ms → ONNX INT8 135ms) |
| INT8 정확도 | FP32 대비 **99.75%** 일치, 모델 1.3GB → 322MB |
| release gate | raw PII 로깅 = 0, invalid offset = 0 (docs/08 §8) |

## 5. 데모 운영 팁 — 주민번호/카드는 "유효한 번호"로

우리 시스템은 **체크섬(검증숫자)까지 확인**해서 **진짜 주민번호/카드만** 잡고, 형식만 비슷한 가짜·오타는 거른다(strict). 이건 **오탐을 줄이는 precision 강점**이다. 단, 데모에서 즉흥으로 아무 13자리나 넣으면 안 잡힐 수 있으니:

- **라이브 시연은 예제 버튼 사용** — `900101-1234568`(RRN), `4532015112830036`(카드)은 체크섬이 맞는 유효 번호라 확실히 잡힌다.
- 교수님이 즉흥으로 임의 번호를 넣어 안 잡히면, 당황하지 말고 **강점으로 설명**:
  > "방금 번호는 체크섬이 맞지 않는 가짜라 일부러 거른 겁니다. 저희는 형식만 보는 게 아니라 검증숫자까지 확인해서 **진짜 PII만** 마스킹하고 오탐을 줄입니다. 유효한 번호는 이렇게 잡힙니다." (예제 클릭)
- 이름·주소(NER)·전화·이메일은 체크섬과 무관하므로 아무 값이나 잘 잡힌다.

## 6. 예상 질문 대비

- **"체크섬 안 맞는 번호는 왜 안 잡히나요?"**
  → 의도된 동작입니다. 형식만 맞는 가짜를 거르고 검증숫자까지 통과한 진짜 PII만 마스킹해서 false positive를 줄입니다(strict). 운영 정책에 따라 warn(형식만 맞으면 저신뢰로 탐지) 모드로 완화할 수도 있습니다(detector config).
- **"텍스트형 PII(연봉/처방명)는 못 잡나요?"**
  → 현재는 구조형(RRN/전화/카드) + NER(이름/주소/조직) 중심. 텍스트형은 키워드 사전 확장이 향후 과제(1차 캡스톤 Layer 0 키워드 사전 자산 있음).
- **"외부 평가는 했나요?"**
  → KLUE-NER test에서 macro-F1 0.766. 최종 unseen 평가(KLUE test + KoJailFuzz 6종)는 W6에 1회 수행 예정(test set 오염 방지 정책).
- **"실서비스 적용 가능한가요?"**
  → ONNX INT8로 CPU 135ms 확보. LiteLLM Gateway 통합 경로 검증됨(1차 캡스톤). 단일 가드레일이 4-detector를 병렬 통합.
- **"왜 직접 fine-tune 했나요?"**
  → 기존 한국어 NER 모델은 라이선스 제약(모두의말뭉치/CC-BY-NC) 또는 라벨 포맷 불일치. klue 기반 fine-tune이 라이선스·성능 모두 우위(v2 실패 경험 포함 6회 실험).
