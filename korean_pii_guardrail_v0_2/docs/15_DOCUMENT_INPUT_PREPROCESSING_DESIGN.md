[가정] 가드레일은 이미 "텍스트 입력"을 받아 검증·마스킹하는 기능이 구현되어 있으며, 본 작업은 그 앞단에 전처리 단계를 추가하는 것



1. 개요
1.1 배경
현재 가드레일은 텍스트 입력만 처리할 수 있다. 사용자가 문서 파일(PDF, 오피스 문서, 이미지 등)을 입력하는 경우, 해당 파일이 가드레일 검증·마스킹을 거치지 못한 채 처리되거나 처리 자체가 불가능하다.
이로 인해 다음 문제가 발생한다.

문서 안에 포함된 민감정보(개인정보, 인증정보 등)가 마스킹 없이 전송될 위험이 있다.
가드레일의 검증 정책이 문서 입력 경로에는 적용되지 않아 정책 사각지대가 생긴다.

1.2 목적
문서 파일이 입력되면 가드레일 앞단(전처리 단계)에서 텍스트로 변환한 뒤, 변환된 텍스트가 기존 가드레일 검증·마스킹 흐름을 그대로 통과하도록 한다. 이를 통해 입력 형태(텍스트/문서/이미지)와 무관하게 동일한 보안 정책을 일관되게 적용한다.
1.3 범위
포함 범위

문서 파일 → 텍스트 추출 모듈 신규 구현 (PDF / 오피스 문서)
이미지·스캔본에 대한 OCR 텍스트 추출
추출된 텍스트를 기존 가드레일 파이프라인에 연결
추출 실패·예외 상황 처리
단위/통합 테스트

제외 범위

가드레일 검증 로직 자체의 변경 (기존 로직 재사용)
마스킹 규칙(정규식·정책) 자체의 신규 정의 (기존 규칙 재사용)
손글씨 인식 등 고난도 OCR (표준 인쇄체 기준으로 한정)


2. 용어 정의
용어정의가드레일(Guardrail)입력 텍스트에 대해 정책 검증과 민감정보 마스킹을 수행하는 기존 모듈전처리 단계가드레일 검증 직전에 문서를 텍스트로 변환하는 신규 단계텍스트 추출문서 파일 바이너리에서 사람이 읽을 수 있는 텍스트를 뽑아내는 처리OCR이미지·스캔본에서 문자를 광학적으로 인식해 텍스트로 변환하는 처리텍스트 레이어PDF 내부에 이미 포함된, 추출 가능한 텍스트 데이터마스킹검출된 민감정보를 **** 등 대체 문자열로 치환하는 처리다운스트림마스킹된 텍스트를 전달받는 다음 처리 단계(예: LLM 호출)

3. 현재 구조 (As-Is)
[텍스트 입력] ──→ [가드레일 검증] ──→ [마스킹] ──→ [다운스트림 전송]

[문서 파일 입력] ──→ ✗ 처리 불가 / 정책 미적용

텍스트 입력만 가드레일을 통과한다.
문서·이미지 입력 경로는 가드레일 정책의 사각지대다.


4. 목표 구조 (To-Be)
[텍스트 입력] ───────────────────────────┐
                                         ├──→ [가드레일 검증] ──→ [마스킹] ──→ [다운스트림 전송]
[문서/이미지 파일 입력] ──→ [전처리:      │
                            문서→텍스트   │
                            변환]  ───────┘
핵심 설계 원칙:

단일 진입점 통일 — 문서든 이미지든 텍스트든 결국 "텍스트"로 정규화되어 동일한 가드레일을 통과한다.
가드레일 로직 무변경 — 가드레일은 입력이 문서였는지 모른다. 전처리 단계가 그 차이를 흡수한다.
마스킹은 가드레일 통과 후 수행 — 변환된 텍스트가 검증을 거친 뒤 마스킹되어 전송된다. (기존 흐름 유지)


5. 전체 처리 흐름
5.1 시퀀스

  │  2-3. 텍스트 추출

  ▼

  │  - 마스킹된 텍스트만 전달

5.2 단계별 책임
단계책임신규/기존[1] 입력 라우터입력 타입 판별 및 분기신규(소규모)[2] 전처리문서·이미지 → 텍스트 변환신규(주요 작업)[3] 가드레일 검증정책 검증기존 재사용[4] 마스킹민감정보 치환기존 재사용[5] 다운스트림 전송결과 전달기존 재사용

6. 컴포넌트 상세 설계
6.1 입력 라우터 (Input Router)
역할: 입력이 텍스트인지 문서/이미지 파일인지 판별하여 흐름을 분기한다.
판별 기준

입력이 파일 객체/바이너리이고 지원 확장자 목록에 포함되면 → 전처리 단계로
그 외 일반 문자열이면 → 기존대로 바로 가드레일로

6.2 전처리 단계 — 문서 텍스트 추출기 (Document Extractor)
본 작업의 핵심 모듈이다. 포맷별 추출기를 두고, 공통 인터페이스로 추상화한다.
6.2.1 구조
DocumentExtractor (퍼사드)
  ├─ PdfExtractor      ─── 텍스트 레이어 없으면 OcrExtractor로 폴백
  ├─ DocxExtractor
  ├─ XlsxExtractor
  ├─ PptxExtractor
  ├─ ImageExtractor    ─── 내부적으로 OcrExtractor 사용
  ├─ OcrExtractor      ─── 이미지·스캔본 공통 OCR 처리
  └─ PlainTextExtractor

각 추출기는 동일한 추상 인터페이스(extract(file) -> ExtractResult)를 구현한다.
DocumentExtractor가 확장자/MIME 타입을 보고 적절한 추출기를 선택한다(전략 패턴).
OcrExtractor는 독립 컴포넌트로 두어 ImageExtractor와 PdfExtractor(폴백 시)가 공유한다.
새 포맷 추가 시 추출기 클래스만 추가하면 되도록 확장 가능하게 설계한다.

6.2.2 추상 인터페이스 (의사코드)
interface BaseExtractor:
    supports(file_meta) -> bool
    extract(file) -> ExtractResult

class ExtractResult:
    text         : string        # 추출된 본문 텍스트
    page_count   : int           # 페이지/시트 수
    char_count   : int           # 글자 수
    extraction_method : enum      # TEXT_LAYER | OCR
    warnings     : list[string]  # 부분 실패 등 경고

extraction_method를 결과에 포함하는 이유: OCR 결과는 텍스트 레이어 추출보다 정확도가 낮을 수 있어, 후속 처리·로그·디버깅에서 출처를 구분할 필요가 있다.

6.2.3 PDF 처리 — 텍스트 레이어 우선, OCR 폴백
PDF는 두 종류로 나뉜다.

텍스트 레이어가 있는 PDF — 텍스트를 바로 추출 (빠르고 정확)
스캔본 PDF (이미지만 있음) — 텍스트 레이어가 없거나 비어 있음

PdfExtractor는 먼저 텍스트 레이어 추출을 시도하고, 결과가 비어 있거나 글자 수가 임계값 미만이면 각 페이지를 이미지로 렌더링한 뒤 OcrExtractor로 폴백한다.
6.2.4 OCR 처리 (OcrExtractor)

입력: 이미지 파일 또는 PDF 페이지 렌더링 이미지
처리: 이미지 전처리(흑백 변환·기울기 보정·노이즈 제거) 후 OCR 엔진 호출
출력: 인식된 텍스트
한국어·영어 혼용 문서를 다루므로 OCR 엔진의 다국어 인식 설정이 필요하다.
OCR 엔진 선택은 9장 및 8장(보안)을 함께 참조하여 결정한다.

6.2.5 텍스트 정규화
추출 직후 다음 정규화를 거친다.

제어 문자·널 바이트 제거
연속 공백·개행 축약
페이지/시트 구분자 삽입 (예: \n--- page 2 ---\n)
OCR 결과의 경우, 인식 오류로 생긴 깨진 문자 패턴을 가능한 범위에서 정리

6.3 가드레일 검증 (기존 재사용)
전처리 단계의 출력 텍스트를 기존 가드레일 검증 함수에 그대로 전달한다. 가드레일 입장에서는 입력이 문서·이미지였다는 사실을 알 필요가 없다.
6.4 마스킹 (기존 재사용)
가드레일 검증 통과 후, 기존 마스킹 모듈이 다음 전 항목을 치환한다.

개인정보: 이름, 주민등록번호, 전화번호, 이메일, 주소 등
인증정보: API 키, 비밀번호, 토큰
사내 기밀: 프로젝트명, 내부 코드/식별자
커스텀 정규식 패턴: 사내 정의 패턴


마스킹 규칙 자체는 본 작업에서 신규 정의하지 않고 기존 규칙을 재사용한다. 다만 OCR 결과 텍스트는 인식 오류로 패턴이 어긋날 수 있어 마스킹이 누락될 위험이 있다. 이 위험과 대응은 8장에 별도로 기술한다.


7. 예외 및 엣지 케이스 처리
상황처리 방안지원하지 않는 확장자명확한 오류 메시지 반환, 처리 중단파일 크기 초과상한선 설정(예: 10MB). 초과 시 거부 — 상한값 리뷰 시 결정 필요손상된/열 수 없는 파일추출 실패 처리, 오류 메시지 반환암호로 보호된 PDF/문서복호화 시도하지 않음. 거부 후 사용자에게 안내텍스트 레이어 없는 스캔 PDFOCR로 폴백 처리OCR 인식 결과 0글자빈 문서 또는 인식 실패로 처리, 경고 반환OCR 정확도 저하 (저해상도·기울어진 이미지)이미지 전처리 적용. 그래도 신뢰도 낮으면 경고를 결과에 포함추출 텍스트 과대 (토큰 한도 초과)청킹 또는 길이 제한 적용 — 정책 리뷰 시 결정 필요다중 인코딩 텍스트 파일인코딩 감지 후 디코딩, 실패 시 오류

8. 보안 고려사항
본 기능은 보안 파이프라인의 일부이므로 아래를 준수한다.
8.1 OCR 엔진 선택과 데이터 유출 위험 ⚠️ (중요)
OCR은 마스킹 이전 단계에서 수행된다. 즉 OCR 엔진에 전달되는 것은 마스킹되지 않은 원본 문서/이미지다.

외부 클라우드 OCR 서비스(예: AWS Textract, Google Document AI, Azure 등)를 사용하면, 민감정보가 포함된 마스킹 전 원본이 외부 사업자에게 전송된다.
이는 민감정보 유출을 막는다는 가드레일의 목적 자체와 충돌한다.
권장: OCR은 온프레미스/사내망 내 엔진으로 처리한다. 부득이하게 외부 OCR을 써야 한다면, 데이터 처리 위탁·보안 검토를 반드시 거치고 별도 승인 항목으로 분리한다.

8.2 일반 보안 사항

마스킹 전 텍스트 비노출 — 추출/OCR 직후의 원문 텍스트(마스킹 전)는 로그·예외 메시지·임시 파일에 남기지 않는다. 디버그 로그에도 본문을 출력하지 않는다.
임시 파일 즉시 삭제 — 추출·OCR 과정에서 생성한 임시 파일(렌더링 이미지 등)은 처리 완료/실패 즉시 삭제한다.
파일 검증 우선 — 확장자뿐 아니라 MIME 타입을 함께 검증해 위장 파일을 거른다.
추출 라이브러리 취약점 관리 — PDF/오피스 파서와 OCR 엔진은 악성 파일 공격면이 될 수 있으므로 버전을 고정하고 정기 업데이트한다.
마스킹 보장 — 어떤 경로로도 가드레일·마스킹을 건너뛴 텍스트가 다운스트림에 전달되지 않도록, 전처리 출력은 반드시 가드레일을 거치게 흐름을 강제한다.

8.3 OCR 인식 오류로 인한 마스킹 누락 위험
OCR은 글자를 100% 정확히 인식하지 못한다. 예를 들어 주민번호의 숫자 하나가 다른 문자로 인식되면, 마스킹 정규식이 패턴을 못 잡아 민감정보가 그대로 통과할 수 있다.
대응 방안 (리뷰 시 정책 결정 필요):

OCR 신뢰도가 낮은 결과는 사용자에게 경고하거나, 보수적으로 처리(차단)하는 정책을 검토
마스킹 정규식을 OCR 오인식에 어느 정도 관대하게 설계할지 검토
본 위험은 완전 제거가 어려우므로 **알려진 한계(known limitation)**로 명시하고 운영 정책에 반영


9. 구현 스택 및 라이브러리

스택 미정 상태이므로, 스택 확정 후 아래에서 선택한다. 추출기 인터페이스는 언어 중립적으로 설계하므로 스택 변경의 영향은 추출기 내부 구현에 한정된다.

9.1 스택별 라이브러리 후보

9.2 스택 무관 대안 — 추출 전용 서비스 분리
스택이 미정이고, 향후 변경 가능성이 있다면 문서 추출을 별도 서비스/모듈로 분리하는 방안을 권장한다.

메인 애플리케이션은 추출 서비스에 파일을 넘기고 텍스트를 돌려받는다.
추출 서비스는 Apache Tika + Tesseract 조합 등으로 구성하면 언어 선택과 무관하게 동작한다.
장점: 메인 스택과 분리, 무거운 파싱/OCR 부하 격리, 교체 용이
단점: 컴포넌트 추가에 따른 운영 복잡도 증가

→ 리뷰에서 (a) 메인 스택에 직접 구현 vs (b) 추출 서비스 분리 중 결정 필요.

10. 성능 고려사항

문서 파싱, 특히 OCR은 텍스트 입력보다 현저히 무겁다. 페이지 수가 많은 스캔 PDF는 처리 시간이 길어질 수 있다.
동기 처리 시 응답 지연이 크므로, OCR 경로는 비동기 처리/작업 큐 도입을 우선 검토한다.
대용량 파일은 메모리 사용량이 크므로 페이지 단위 스트리밍 처리를 우선 검토한다.
포맷별·OCR 평균 소요 시간을 측정해 타임아웃 기준을 정한다.

11.3 테스트 데이터

포맷별 정상 샘플 + 의도적으로 손상시킨 파일
