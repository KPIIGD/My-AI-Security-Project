• *요구사항 정리*
회의록, Slack 대화, GitHub issue를 읽고 “무엇을 만들어야 하는지” 정리합니다.
예를 들어 “로그인 오류를 고쳐야 함”이라는 말만 있으면 부족하니, AI가 acceptance criteria, edge case, 필요한 파일 후보를 정리할 수 있습니다.
• *설계 초안*
API 구조, 데이터 흐름, 아키텍처 선택지를 비교해줍니다.
단, 최종 설계 결정은 사람이 해야 합니다. AI는 그럴듯한 설계를 만들 수 있지만, 제품 책임이나 장기 유지보수 판단은 사람이 가져가야 합니다.
• *구현*
작은 버그 수정, 반복적인 CRUD, glue code, migration script 같은 작업에 잘 맞습니다.
중요한 점은 작업을 작게 쪼개야 한다는 것입니다. “서비스 전체 개선해줘”보다 “이 validation 버그를 테스트와 함께 고쳐줘”가 훨씬 안전합니다.
• *리팩터링*
죽은 코드 제거, 함수 분리, 타입 정리, 이름 개선 같은 작업을 할 수 있습니다.
다만 리팩터링은 동작을 바꾸면 안 되므로 기존 테스트가 중요합니다.
• *테스트 작성*
AI는 unit test, integration test, regression test 초안을 잘 만듭니다.
하지만 테스트가 정말 버그를 잡는지는 확인해야 합니다. 통과만 하는 의미 없는 테스트를 만들 수도 있습니다.
• *코드 리뷰 보조*
PR 요약, 위험한 파일 표시, reviewer checklist 작성에 쓸 수 있습니다.
하지만 approval은 사람이 해야 합니다. AI가 “문제 없음”이라고 해도 최종 책임자는 사람이기 때문입니다.
• *보안 점검 보조*
secret scanning, dependency review, threat model checklist 작성에 도움됩니다.
단, 고위험 finding은 사람이 직접 판단해야 합니다.
• *문서화*
README, runbook, API docs, changelog, onboarding guide를 정리할 수 있습니다.
여기서 중요한 건 실제 명령어와 최신 경로를 검증하는 것입니다. AI가 오래된 명령을 그럴듯하게 쓸 수 있습니다.
• *온보딩*
새 팀원이 저장소 구조, 빌드 방법, 첫 작업을 이해하도록 repo map을 만들 수 있습니다.
• *PR/CI 자동화*
 CI 실패 로그를 읽고 작은 수정 PR을 만들 수 있습니다.
 하지만 protected branch와 required checks를 반드시 유지해야 합니다.
• *UI 검증*
 Playwright, screenshot diff, browser automation으로 화면이 깨졌는지 확인할 수 있습니다.
 최종 UX 판단은 사람이 해야 합니다.
• *운영 자동화*
 release note, incident summary, Slack digest, metrics report 작성에 쓸 수 있습니다.
 하지만 production write 권한은 주면 안 됩니다.

LLM에게 프로젝트 이해를 시키는 방법
필수 컨테스트 계층(ai에게 주는 프로젝트 설명서 묶음)
• 이 프로젝트가 무엇인지
• 어떤 폴더를 먼저 봐야 하는지
• 어떤 규칙을 절대 어기면 안 되는지
• 테스트는 어떻게 돌리는지
• 완료 기준이 무엇인지
• 건드리면 안 되는 파일이나 범위가 무엇인지
1.`AGENTS.md`, `CLAUDE.md`, `GEMINI.md` 같은 *고정 규칙 파일*
AI가 항상 따라야 할 행동 규칙입니다. 예를 들어 “민감정보를 로그에 남기지 마라”, “테스트를 실행해라”, “이 폴더는 reference only다” 같은 규칙이 들어갑니다.
2.`docs/index.md`, `llms.txt`, repo map 같은 *문서 지도*
AI에게 “어디를 먼저 읽어야 하는지” 알려주는 지도입니다. 프로젝트가 커지면 파일이 많아서 AI가 엉뚱한 파일을 읽을 수 있습니다. 문서 지도가 있으면 탐색 비용이 줄어듭니다.
3. architecture docs, interface spec 같은 *설계 원천*
코드를 왜 이렇게 만들었는지 설명하는 문서입니다. AI가 임의로 구조를 바꾸지 않게 해줍니다.
4. ADR, `docs/decisions/` 같은 *결정 기록*
과거에 어떤 선택을 했고 왜 그렇게 결정했는지 남기는 곳입니다. 예를 들어 “RAG는 이번 버전에 넣지 않는다” 같은 결정이 있으면 AI가 다시 RAG를 구현하려 하지 않게 됩니다.
5. OpenAPI, JSON Schema, config schema 같은 *계약 원천*
API나 데이터 형식을 AI가 추측하지 않게 해줍니다. 계약 문서가 없으면 AI가 그럴듯하지만 틀린 필드명을 만들 수 있습니다.
6. tests, CI workflow, runbook 같은 *검증 원천*
작업이 끝났는지 판단하는 기준입니다. “잘 된 것 같아요”가 아니라 테스트와 CI로 확인해야 합니다.


지식관리 도구
1. Obsidian
AI 개발 자동화에서는 이런 용도로 좋습니다.
• 실험 로그
• 프롬프트 기록
• 평가 결과
• threat model
• 결정 기록
• 프로젝트별 연구 노트
예를 들어 “왜 이 AI 도구를 선택했는가”, “어떤 실험에서 실패했는가”, “어떤 프롬프트가 잘 먹혔는가” 같은 내용을 남기기 좋습니다.
문서가 제안하는 구조도 이런 식입니다.
```AI-Security-KB/
  00-inbox/
  10-projects/
  20-concepts/
  30-sources/
  40-agent-context/
  60-templates/
  70-dashboards/
  90-archive/```
초보자식으로 말하면 Obsidian은 *AI 개발 프로젝트의 연구 노트장*입니다.

2. Slack
• Slack에서는 현재 작업 흐름과 알림을 관리한다.
• 중요한 결정은 `docs/decisions/`, Google Docs, Obsidian 같은 durable 문서로 옮긴다.
• AI에게는 필요한 thread나 채널만 제한적으로 검색하게 한다.
• OAuth scope, app approval, audit log, DLP를 같이 설계한다.
AI에게 현재 무슨 일이 벌어지고 있는지 알려주는 실시간 알림판


3. Google Drive, Docs, Sheets, Slides
• Drive shared drive: 공식 파일 저장소, 권한, 버전 이력
• Docs: 요구사항, 회의록, 결정, 보고서
• Sheets: 실험 결과, 평가표, 비용 추적
• Slides: 발표자료
• Gemini for Workspace: 문서 요약, 초안, 검색 보조
• NotebookLM: 선별한 자료 기반 질의응답과 출처 확인
Google Drive는 *팀의 공식 서류함,공식 산출물*
보고서, 회의록, 발표자료처럼 나중에 제출하거나 검토할 자료는 Drive/Docs

아키텍처
```Slack 요청/장애/리뷰 호출
  -> GitHub Issue / Project item
  -> 사람 또는 coding agent가 branch 작업
  -> Pull Request
  -> CI: test, lint, typecheck, security scan
  -> CODEOWNERS / required reviewer / human review
  -> staging/prod environment approval
  -> deploy 결과와 incident 알림은 Slack
  -> 결정, 회의록, 승인 문서는 Docs/Drive 또는 Notion에 링크```
1. *Slack에서 요청이 생김*
예를 들어 “로그인 오류가 납니다”, “이 기능 추가해주세요”, “배포 후 알림이 이상합니다” 같은 말이 나옵니다.
2. *GitHub Issue로 바꿈*
Slack 대화는 흘러가기 쉽기 때문에, 실제 작업 단위는 GitHub Issue로 만듭니다.
Issue에는 목표, 범위, 완료 기준, 참고 링크를 적습니다.
3. *사람 또는 coding agent가 branch에서 작업*
AI agent가 바로 main 브랜치를 고치면 위험합니다.
별도 branch에서 작업해야 합니다.
4. *Pull Request 생성*
변경사항은 PR로 모읍니다. PR은 코드 변경, 설명, 테스트 결과, 리뷰 코멘트를 한곳에 모으는 단위입니다.
5. *CI 검사 실행*
테스트, lint, typecheck, 보안 검사, dependency scan이 자동으로 돌아갑니다.
AI가 “됐습니다”라고 말해도 CI가 실패하면 완료가 아닙니다.
6. *사람 리뷰*
CODEOWNERS나 required reviewer가 리뷰합니다.
중요한 점은 approval은 AI가 아니라 사람이 해야 한다는 것입니다.
7. *staging/prod 승인 분리*
실제 배포는 별도 승인 절차를 둡니다.
특히 production DB, cloud admin, 배포 권한은 agent에게 주면 안 됩니다.
8. *결과와 결정 기록*
배포 결과나 장애 알림은 Slack으로 공유하고, 중요한 결정은 Docs/Drive/Notion 또는 `docs/decisions/`에 남깁니다.
이 구조의 핵심은 *역할 분리*입니다.
• GitHub: 코드, 이슈, PR의 원장
• Slack: 알림과 실시간 대화
• Drive/Docs: 공식 산출물과 승인 문서
• Obsidian/Notion: 지식 베이스와 실험 기록
• Coding agent: branch/PR 작업자
• CI/CD: merge/deploy gate
• 사람: 최종 판단과 승인
